# ADR-005: Use FAISS for Vector Similarity Search

**Date:** 2024-01-01  
**Status:** Accepted

---

## Context

Semantic search requires vector similarity lookups on multilingual text embeddings generated by Vertex AI (`text-multilingual-embedding-002`). Each embedding is a high-dimensional float vector; at query time the system must find the top-k most similar resume vectors to a query embedding.

The target scale is approximately 1,000–100,000 resumes. Requirements are:

- Low latency for similarity queries (< 200 ms)
- No additional hosting cost at baseline scale
- Python-native — all AI/data services are Python
- Suitable for in-process operation inside a Cloud Run container

## Decision

The Search Base service manages an in-process FAISS (Facebook AI Similarity Search) index for semantic resume search. FAISS runs directly inside the Python process — no external search service is required.

The FAISS index is optionally persisted to a mounted volume (`FAISS_INDEX_PATH`) so it survives container restarts. If no persisted index is found on startup, the Search Base rebuilds it by fetching all embeddings from Firestore.

## Alternatives Considered

**Vertex AI Matching Engine (managed Vector Search):** Fully managed, horizontally scalable vector search. Rejected because it has significant per-index hosting costs (minimum ~$65/month for a deployed index) that conflict with the low-cost design goal at the target scale.

**Pinecone:** Managed vector database with a generous free tier. Rejected because it is not a GCP-native service, adding an external dependency and separate billing. At the target scale, FAISS in-process is sufficient.

**pgvector (PostgreSQL extension):** Vector similarity built into PostgreSQL. Rejected because the project uses Firestore (not PostgreSQL) as its primary database. Introducing a PostgreSQL instance solely for vector search adds unnecessary operational overhead.

## Consequences

- **Easier:** No external vector search service to provision or pay for.
- **Easier:** Python-native — integrates directly with the AI Worker and Search Base Python services.
- **Easier:** Low latency for in-process searches at the target scale.
- **Harder:** The FAISS index is in-memory — it is lost on pod/container restart unless persisted to a volume.
- **Harder:** No managed SLA; the Search Base must implement its own index rebuild and health-check logic.
- **Harder:** Search Base must set `--min-instances=1` on Cloud Run to avoid cold starts that would require a full index rebuild before the first query.
- **Follow-on:** A `POST /api/v1/index/rebuild` endpoint on Search Base should be provided to allow manual index rebuilds. See [Troubleshooting](../troubleshooting.md#faiss-index-issues).
